## TL;DR

When a user reloads (or navigates to) a chat with prior messages persisted in `conversation_event`, the chat composer renders empty and the historical messages never appear in the sidebar conversation. Sending a new message starts a fresh turn with no connection to the prior history. The root cause is most likely a payload-shape mismatch between what we persist and what `eve/react`'s `defaultMessageReducer` expects to replay through `useEveAgent({ initialEvents })`.

This spec is split in two: **Phase 1 â€” diagnosis** (must run first; tells us which of the 4 candidate causes is actually in play), and **Phase 2 â€” fix** (the right combination of changes once the data is in hand).

---

## Context

`PR #6` ships the chat sidebar with a `conversation_event` log + an `useEveAgent({ initialEvents, initialSession })` wiring on the client. The intent: on reload, the user sees the prior messages without waiting for a network round-trip.

In local testing, after a refresh the chat shows only the empty composer. Sending a new message starts a turn that does not include the prior history. The persisted events exist in the DB; the reducer does not project them into `EveMessage[]`.

A close reading of the code surfaces **four overlapping candidate causes**, each independently sufficient to produce the symptom. The fix is the union of the remedies, but the exact shape depends on which causes are actually triggered â€” hence the diagnostic-first structure.

## Phase 1 â€” Diagnosis (no code changes)

Run these checks against the deployed DB (`DATABASE_URL=postgres://neondb_owner:...@ep-blue-cake-aw085mwo-pooler.c-12.us-east-1.aws.neon.tech/neondb`) and a local browser session. The output tells us which of the 4 candidates to prioritize.

### Check 1 â€” Are events persisted at all?

```sql
SELECT count(*) FROM conversation_event WHERE conversation_id = '<id-of-a-chat-with-history>';
```

- **0 rows** â†’ the batcher is silently failing (network, off-guard 4xx, rejected events). Check Check 2 anyway to confirm.
- **N rows** â†’ events are reaching the DB. Proceed.

### Check 2 â€” What is the persisted `payload` shape?

```sql
SELECT type, payload
FROM conversation_event
WHERE conversation_id = '<id>'
ORDER BY id
LIMIT 5;
```

We expect to see two possible shapes, depending on what the batcher wrote:

| Expected by current code | Written by current code |
|---|---|
| `{ type: "...", data: {...}, meta: {...} }` (a `HandleMessageStreamEvent`) | `{ type: "...", sequence: 0, payload: { data: {...}, meta: {...} } }` (an `IncomingEvent` with a wrapped payload) |

If the column shows the **envelope** (`payload: { data, meta }`), then the reconstruction in `agent-chat.tsx` is mandatory on read-back â€” and the reconstruction has a bug (see candidate #1).

### Check 3 â€” What is the `sequence` of every event?

```sql
SELECT id, type, sequence
FROM conversation_event
WHERE conversation_id = '<id>'
ORDER BY id;
```

We expect `sequence: 0` for every row. The reducer does not depend on `sequence` per se, but the list view does `ORDER BY sequence` â€” so all-zero values make the replay order **non-deterministic** (Postgres heap order, not stream order).

### Check 4 â€” What does the browser console say on reload?

Open DevTools on `/login` â†’ sign in â†’ navigate to `/?c=<id>`. Watch for:
- Network: does `GET /api/conversations/<id>/events?since=0&limit=100` return rows?
- Network: does the response carry events with the expected `type` values?
- JS console: any thrown error from `useEveAgent` / the default reducer?
- React DevTools: what does `EveAgentStore.data` look like after mount (specifically `messages.length`)?

A clean stream with no errors and `messages.length === 0` confirms the reducer is the issue, not the wire.

### Decision matrix

| Check 1 | Check 4 (messages.length) | Likely cause | Phase 2 step |
|---|---|---|---|
| 0 rows | n/a | Batcher silently failing | Step 1 only |
| N rows | 0 | Reducer can't replay | Steps 2 + 3 |
| N rows | > 0 | Something else (UI hides them) | Investigate `isEmpty` |

In practice, the four candidates below are all real and worth fixing together â€” the cost of doing all four is small (~80 LoC), and the diagnostic phase confirms the order of priority.

## Candidate causes

### Candidate #1 â€” `agent-chat.tsx` reconstruction is fragile

In `app/_components/agent-chat.tsx` (current code), the initial events are reconstructed like so:

```ts
const initialEveEvents = useRef(
  initialEvents.map((e) => {
    const { data, meta } = (e.payload ?? {}) as { data?: unknown; meta?: { at?: string } };
    return {
      type: e.type,
      ...(data !== undefined ? { data } : {}),
      ...(meta ? { meta } : {}),
    } as unknown as HandleMessageStreamEvent;
  }),
);
```

Two problems with this approach:

- The reconstruction is **a custom re-shaping** that diverges from the wire format. Any future event whose original shape includes more than `{ type, data, meta }` (e.g., a `providerMetadata` field added in a future eve version) is silently lost.
- The `as unknown as HandleMessageStreamEvent` cast hides the mismatch from the type system. A single field rename in eve would compile cleanly and break the replay without a runtime signal.

**Fix:** do not reshape at all. Persist the full `HandleMessageStreamEvent` JSON in `payload`, and replay it as-is.

### Candidate #2 â€” Batcher wraps the event in a synthetic envelope

In `app/_components/agent-chat-batcher.ts` (current code):

```ts
function toIncomingEvent(event: EveRawEvent) {
  return {
    type: raw.type,
    sequence: 0,
    payload: { data: raw.data ?? null, meta: raw.meta ?? null },
  };
}
```

The `payload: { data, meta }` envelope is the source of the reconstruction in #1. Removing it is the cleaner fix.

**Fix:** send the raw event (drop the envelope).

```ts
function toIncomingEvent(event: EveRawEvent) {
  return {
    sequence: 0, // server assigns the real one
    payload: event as unknown as Record<string, unknown>,
  };
}
```

### Candidate #3 â€” `sequence: 0` everywhere â†’ non-deterministic replay order

In `lib/conversations.ts#appendEvents`, the `sequence` is taken from the client (which always sends `0`). The DB then has `sequence: 0` for every row. `listEvents` orders by `sequence`, which is non-deterministic.

The reducer does not strictly require a monotonic sequence, but **a missing `message.received` before its `message.appended` events** will silently fail to construct the message.

**Fix:** assign the `sequence` server-side, in the route, with a per-conversation row lock to serialize concurrent batches.

```ts
// in appendEvents, inside a transaction:
const { rows: [{ max }] } = await tx.execute(
  sql`SELECT COALESCE(MAX(sequence), 0) AS max FROM ${conversationEvent}
      WHERE conversation_id = ${conversationId} FOR UPDATE`,
);
let next = Number(max) + 1;
for (const evt of accepted) {
  evt.sequence = next++;
}
```

Alternative: add a `BIGSERIAL` global sequence and use it. Rejected: less locality, harder to debug.

### Candidate #4 â€” The reducer requires events the batcher does not capture

The eve `defaultMessageReducer` (verified against `eve/dist/src/client/message-reducer.js`) matches the optimistic `client.message.submitted` event to the server `message.received` event by `submissionId` on the optimistic event. On a fresh hook instance (replay), no `client.message.submitted` exists in the events list â€” but the reducer code path falls back to processing `message.received` directly via `#w` instead of `#T` in that case. So in theory this is fine.

**Likely not a real cause in v1** â€” but worth confirming via Check 4. If `messages.length === 0` after Check 1 returns N rows and Check 2 shows valid shapes, then candidate #4 is **not** the cause, and the fix is purely #1 + #2 + #3.

## Step-by-step implementation

### Step 1 â€” Drop the envelope in the batcher

`app/_components/agent-chat-batcher.ts`, replace `toIncomingEvent` with:

```ts
function toIncomingEvent(event: EveRawEvent): { type: string; payload: unknown } {
  return { type: event.type, payload: event };
}
```

The `IncomingEvent` type in `lib/conversations.ts` gains a `type: string` field that's the event-type discriminant (not the wire envelope anymore).

### Step 2 â€” Server-side sequence assignment

`lib/conversations.ts#appendEvents`, wrap the existing `accepted.push(...)` loop in a transaction that locks the conversation row, queries `MAX(sequence)`, and assigns `next++` per event:

```ts
import { sql } from "drizzle-orm";

export async function appendEvents(userId, conversationId, events) {
  // ... existing validation, unchanged ...
  
  if (accepted.length > 0) {
    await db.transaction(async (tx) => {
      // Lock the conversation row to serialize concurrent batches.
      const [{ max }] = await tx.execute(
        sql`SELECT COALESCE(MAX(sequence), 0) AS max
            FROM ${conversationEvent}
            WHERE conversation_id = ${conversationId}
            FOR UPDATE`,
      );
      let next = Number(max) + 1;
      for (const evt of accepted) {
        evt.sequence = next++;
      }
      await tx
        .insert(conversationEvent)
        .values(accepted)
        .onConflictDoNothing({ target: [conversationEvent.conversationId, conversationEvent.eventId] });
    });
  }
  return { accepted: accepted.length, rejected };
}
```

`db.transaction` is the existing Drizzle API. The `FOR UPDATE` is the row lock; without it two concurrent batches can read the same `MAX` and clobber.

### Step 3 â€” Replay the events without reshaping

`app/_components/agent-chat.tsx`, replace the `initialEveEvents` `useRef` with a plain type assertion:

```ts
const initialEveEvents = useRef<readonly HandleMessageStreamEvent[]>(
  initialEvents.map((e) => e.payload as HandleMessageStreamEvent),
);
```

The shape is now a 1:1 round-trip â€” what eve emits is what we persist and what we replay.

### Step 4 â€” Backfill (only if Phase 1 Check 2 shows the envelope in existing rows)

```sql
-- db/migrations/0002_backfill_event_shape.sql
UPDATE conversation_event
SET payload = payload->'payload'
WHERE payload ? 'payload'
  AND jsonb_typeof(payload->'payload') = 'object';
```

This un-wraps the envelope. Once Check 2 shows new rows have the flat shape, the WHERE clause stops matching and the migration becomes a no-op on subsequent deployments.

### Step 5 â€” Replay test

`tests/agent-chat-replay.test.ts`, in two layers:

1. **Pure reducer test**: import the `defaultMessageReducer` from `eve/react` (re-exported from `eve/dist/src/client/message-reducer.js`), call `reduce(initial(), event)` for each persisted event, assert the projection has at least one `EveMessage` after a `message.received`.

2. **End-to-end test**: persist 5 events through `appendEvents`, fetch via `listEvents`, replay through the reducer, assert the projection is non-empty and the messages are in the right order.

The first layer is the regression test that catches the root cause; the second is the integration guarantee that the round-trip wire works.

### Step 6 â€” Smoke + integration re-run

Run the existing 22 tests + the new replay test. All must pass. Run `npm run build`. Commit on the impl branch.

## Open questions

These will be resolved during Phase 1 (diagnosis), not implementation:

- **Does the DB have rows at all?** â€” Check 1 will tell us. If 0 rows, candidate #1/2/3 are not the cause; the batcher is failing earlier.
- **What is the persisted `payload` shape?** â€” Check 2 will tell us. If the envelope is in DB, Step 4 is required; if not, Step 4 is skipped.
- **What does the reducer do on the reconstructed event?** â€” Check 4 will tell us. If `messages.length > 0` after the fix, the bug is in the wire shape; if still 0, the reducer needs a custom implementation (out of scope for this spec â€” would justify a follow-up).
- **Concurrent batches on the same conversation** â€” Step 2's `FOR UPDATE` lock serializes them. The cost is a row-level lock for the duration of the insert; acceptable for v1.

## Risks / edge cases

- **Backfill is a one-shot, no rollback** â€” if Step 4 fires, the `UPDATE` is irreversible. Mitigation: run Check 2 first; the migration includes a `WHERE payload ? 'payload'` predicate that is a no-op if the column is already in the new shape.
- **Server-side sequence under concurrent load** â€” the `FOR UPDATE` lock scopes the contention to one conversation, not the whole table. With 1k users, each on a different conversation, the locks do not block each other.
- **Reducer expects events the batcher does not persist** â€” candidate #4. Mitigated by the Phase 1 check; if it triggers, the follow-up is a custom reducer.
- **Stream replay vs `initialEvents` divergence** â€” eve's hook was designed for live streams. Persisted events are an emulation. If eve's reducer changes between versions, the replay breaks. Mitigation: pin eve in `package.json` (already pinned at 0.24.4) and add a smoke test that re-runs the reducer on a fixture event log.
- **Old conversations with the envelope still in DB** â€” handled by the backfill migration in Step 4.
