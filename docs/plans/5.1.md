## TL;DR

The reported symptom ("no history on reload") is reproducible, but the root cause is **not** in the eve reducer, the `useEveAgent` hook, or the event-reconstruction code. Two new unit tests (`tests/replay-diagnosis.test.ts`, `tests/use-eve-agent-replay.test.ts`) prove the full chain â€” reducer + React glue â€” produces the correct 2 messages from the 10 events we pulled live from the DB. The bug is therefore either (a) the page not actually receiving the events (`/?c=<id>` falls through to the empty state) or (b) a per-conversation state issue (the conversation has 0 events in the DB).

This spec is restructured to reflect the evidence: a tight **diagnostic flow** the user runs in their dev session, then a **small fix** once we know which branch is in play.

## What the evidence says (this section is the new structure)

### Evidence 1 â€” `defaultMessageReducer` works on the persisted events

`tests/replay-diagnosis.test.ts` runs the actual `defaultMessageReducer` from `eve/react` against the 10 events we pulled live from the Neon DB. Result:

- **Heap order (current `listEvents` order, all `sequence=0`)** â†’ 2 messages
- **Stream order (logical)** â†’ 2 messages

Both produce `{ id: "turn_0:user", role: "user", parts: [{text: "heyyy", state: "done"}] }` and `{ id: "turn_0:assistant", role: "assistant", parts: [step-start, {text: "Hey there! How can I help you today?", state: "done"}], metadata: { status: "complete" } }`.

**Order does not matter** for the final reducer state, because the reducer is idempotent over the event set (each event's effect is deterministic and `upsertMessage`/`upsertPart` key on stable ids). This eliminates candidate #3 (the `sequence: 0` order issue) as a bug â€” it's still a hygiene problem worth fixing for predictability, but not the cause of the missing history.

### Evidence 2 â€” `useEveAgent` returns the right data

`tests/use-eve-agent-replay.test.ts` mounts `useEveAgent` exactly the way `agent-chat.tsx` does, with the reconstructed events from the DB. Result:

- `agent.data.messages.length === 2` âś“
- `agent.data.events.length === 10` âś“
- `agent.session` carries the cursor (`sessionId`, `continuationToken`, `streamIndex`) âś“
- `agent.status === "ready"` âś“

The hook works. The reconstruction in `agent-chat.tsx` produces events that the hook accepts and projects. This eliminates candidate #1 (reconstruction fragility) and candidate #2 (envelope wrap).

### Evidence 3 â€” The bug is in the page flow, not the data layer

The only remaining possibilities are:

1. **The URL doesn't carry `?c=` on reload.** The page (`app/(authenticated)/(with-sidebar)/page.tsx`) only fetches events when `params.c` is truthy. Without `?c=`, the `if (resolvedId)` block is skipped, `initialEvents` stays `[]`, and `AgentChat` renders an empty composer.
2. **The conversation has 0 events in the DB** (sidebar row was created but no message was ever sent). The page fetches an empty list; the composer is shown.
3. **A real but unobserved bug** (e.g., `listEvents` returns events that the route strips before sending, or the events POST is failing silently for the user's specific session). This is testable in 30 seconds (see Phase 1 below).

## Phase 1 â€” 30-second user-side diagnostic

Before touching any code, run these three checks in the user's dev session.

### Check A â€” Is the URL correct?

Open the browser DevTools console on the affected page and run:

```js
console.log("URL search:", new URLSearchParams(location.search).get("c"));
console.log("Pathname:", location.pathname);
```

If `URL search: <id>` and `Pathname: /` â†’ the URL is correct, the bug is downstream.
If `URL search: null` or `Pathname: /login` â†’ the redirect from `(authenticated)/layout.tsx` is firing (cookie missing?), or the conversation id is being lost in navigation.

### Check B â€” Are events persisted for this conversation?

From the project root, with the dev DB URL:

```sql
SELECT count(*)
FROM conversation_event
WHERE conversation_id = '<the id from Check A>';
```

- **0 rows** â†’ the conversation has no events. Most likely the user opened the chat in the sidebar but never sent a message, OR the batcher POST is failing (silently). See Check C.
- **N > 0 rows** â†’ the events ARE in the DB. The page should be passing them to `AgentChat`. If the user still doesn't see them, the bug is in the page (e.g., a stale build, or `listEvents` is throwing, or the props are not getting through).

### Check C â€” Is the batcher firing?

In the browser DevTools Network tab, on a fresh chat:

1. Open `/?c=<id>` where `<id>` is an existing conversation
2. Send a new message
3. Filter the Network tab by `POST /api/conversations/`
4. You should see: `POST /api/conversations/<id>/events` (batch POST) and `POST /api/conversations/<id>/eve-state` (cursor PATCH)

If these calls **succeed** (200), events are flowing. Reload â€” Check B should now show rows.
If they **fail** (4xx/5xx), inspect the response body. The likely candidates are:
- 401 (cookie missing) â€” the route's `requireUser()` rejected the call
- 413 (batch too large) â€” only if the user sent a huge attachment
- 403 (cross-origin) â€” only if the test env has a different `NEXT_PUBLIC_BETTER_AUTH_URL`

## Step-by-step implementation

### Step 1 â€” Fix the batcher flow (only if Check C is the issue)

If Check C shows the POST failing, the response body has the answer. Common fixes:

- **401 (unauthorized):** the `useEveAgent` `onEvent` is firing after the cookie has been swapped (e.g., on a tab refresh). Wrap the `fetch` in `agent-chat-batcher.ts` in a try/catch that, on 401, requests a session refresh and retries once.
- **413 (batch too large):** reduce `FLUSH_MAX_EVENTS` from 50 to 10 and `FLUSH_INTERVAL_MS` from 500 to 250 in the batcher.
- **403 (forbidden):** the `NEXT_PUBLIC_BETTER_AUTH_URL` env is misconfigured. Compare the `Origin` header on the request with the env value.

If Check C shows 200 OK, the bug is downstream of the route. Jump to Step 2.

### Step 2 â€” Fix the page â†” AgentChat props flow (only if Check B shows N > 0)

Verify the SSR pass is actually reaching `<AgentChat>` with the right `initialEvents`. The simplest diagnostic: add a `console.log("agent-chat: initialEvents.length =", initialEvents.length)` at the top of `AgentChat` in the local dev build, then reload. If the log shows `0`, the page is stripping them before they reach the client. If it shows the expected count, the bug is inside `AgentChat` (despite the unit tests, which would mean a production-only issue â€” likely a React 19 / Next.js 16 hydration mismatch).

If it's a hydration mismatch: the most likely cause is `useRef` initialization running twice in Strict Mode. The first call sets `.current`, the second call would re-initialize â€” but the way `useRef` is defined in React, the initializer runs every render, but `.current` is only set on the first call. So this should be fine. If it isn't, switch to a `useState` instead of `useRef` for the `initialEveEvents` value, computed in a `useState(() => initialEvents.map(...))` so the map runs once.

### Step 3 â€” Make `/` resolve to the most-recent conversation (if Check A shows no `?c=`)

Update `app/(authenticated)/(with-sidebar)/page.tsx`:

```ts
if (!resolvedId) {
  // No ?c= â†’ find the most-recent non-archived conversation and redirect.
  const items = await listConversations(auth.userId, { limit: 1 });
  if (items.length > 0) {
    redirect(`/?c=${items[0].id}`);
  }
  // No conversations: fall through to the empty composer.
}
```

This is the deferred decision from `docs/plans/5.md`. Implementing it makes the URL-as-truth pattern user-visible: reload `/` lands on the latest chat, which is the natural expectation.

### Step 4 â€” Auth cookie loss (if `?c=` redirects to `/login`)

This is the most concerning scenario. It means the `proxy.ts` cookie check or the `(authenticated)/layout.tsx` `getSession` call is failing on reload. Two possible causes:

- **The `betterAuthFn` adapter in `lib/eve-auth.ts` runs before the session is restored.** On a fresh load, the eve channel's auth list runs as part of the agent's startup, and a failed `getSession` there could 401 the rest of the request. Fix: ensure `betterAuthFn` catches and returns `null` instead of throwing.
- **The session cookie is `SameSite=Strict` (or unset) and the reload is a cross-site navigation.** This would set the cookie but the next request wouldn't send it. Fix: in `lib/auth.ts` (the `betterAuth` config), ensure `advanced.useCookies.cookieOptions.sameSite` is `"lax"` â€” the README says it is, but verify in DevTools â†’ Application â†’ Cookies that the value matches.

## Risks / edge cases

- **Step 3's redirect loop risk**: if `listConversations` is slow, every page load on `/` becomes a redirect chain. Mitigation: cache the lookup for the request lifetime, or use `unstable_cache` (Next.js 16 API) to memoize per-user.
- **Step 1's batcher fix could regress turn-end events**: any change to the batcher's retry logic must be tested with a real conversation flow (not just the unit tests). The two regression tests on the branch already cover the most common path.
- **Hydration mismatch in Step 2**: if the fix is `useState` instead of `useRef`, the initial render must not show a "loading" state for the messages. The current code shows them immediately; verify the fix preserves that.
