## TL;DR

Add an inline rename action to the sidebar conversation list — click the title (or a hover-revealed pencil icon) → input field → save on Enter / blur → PATCH the conversation. The PATCH endpoint already exists (`PATCH /api/conversations/:id` accepts `{ title }` via `updateConversation` in `lib/conversations.ts`), so the work is purely UI plus a thin client-side mutation wired into the existing TanStack Query cache. When the user has never set a title, the first user-text part of the conversation auto-derives one (server-side, on PATCH of the conversation).

---

## Context

The data layer for renaming already exists: `app/api/conversations/[id]/route.ts#PATCH` accepts `{ title }` and `lib/conversations.ts#updateConversation` writes it (with a whitelisted field check, so the client can't sneak in `eveSessionId` etc.). The sidebar's `SidebarItem` already carries `title` and the list is already sorted by `pinned DESC, updatedAt DESC`, so renaming a chat naturally re-orders it (the new `updatedAt` from the PATCH bumps it to the top of the inactive group).

What's missing is the UI: today the title is read-only in the sidebar (`item.title ?? item.preview ?? "New chat"`), and a chat that was created with `title: null` shows the generic fallback forever.

---

## Scope

**IN:**
- Inline edit in the sidebar item — click the title (or a hover-revealed pencil icon) → swaps to `<input>` → save on Enter or blur → cancel on Escape
- PATCH call to `/api/conversations/:id` with `{ title }`
- Optimistic update via TanStack Query's `setQueryData` (sidebar shows the new title immediately; rolls back on error)
- Auto-derive title from the first user-text part when the user hits rename on a chat with `title: null` AND the conversation has events. The derivation happens client-side: read the first `message.received` event's `data.parts[0].text`, truncate to 80 chars, send that as the new title
- Add a hover-revealed pencil icon to indicate the action is available
- Empty-title fallback in the sidebar updates: when `title` is null, show "New chat" but the row is still editable (clicking opens the input pre-filled with the derived title or empty)

**OUT (deferred to follow-ups):**
- AI-suggested titles ("Summarize this conversation in 5 words")
- Per-message rename
- Bulk rename
- Server-side title derivation (we do it client-side; if the user opens the rename UI on a chat with 0 events, we just send an empty string or a manual one)
- A separate "edit conversation" page (inline only)

## Step-by-step implementation

### Step 1 — Add a `renameMutation` to `ChatSidebar`

In `app/(authenticated)/_components/sidebar.tsx`, alongside the existing `createMutation`:

```ts
const renameMutation = useMutation({
  mutationFn: async ({ id, title }: { id: string; title: string }) => {
    const res = await fetch(`/api/conversations/${id}`, {
      method: "PATCH",
      headers: { "content-type": "application/json" },
      body: JSON.stringify({ title }),
    });
    if (!res.ok) throw new Error("Failed to rename");
    const { item } = (await res.json()) as { item: SidebarItem };
    return item;
  },
  onMutate: async ({ id, title }) => {
    await qc.cancelQueries({ queryKey: CONVERSATIONS_QUERY_KEY });
    const previous = qc.getQueryData<SidebarItem[]>(CONVERSATIONS_QUERY_KEY);
    qc.setQueryData<SidebarItem[]>(CONVERSATIONS_QUERY_KEY, (old) =>
      (old ?? []).map((i) => (i.id === id ? { ...i, title } : i)),
    );
    return { previous };
  },
  onError: (_err, _vars, ctx) => {
    if (ctx?.previous) qc.setQueryData(CONVERSATIONS_QUERY_KEY, ctx.previous);
  },
});
```

Same optimistic pattern as `createMutation`. The `updatedAt` is not optimistically bumped (the server returns the new row, we use it on success). This avoids a one-frame re-sort jitter.

### Step 2 — Add the `useState` for which row is being edited

```ts
const [editingId, setEditingId] = useState<string | null>(null);
const [draftTitle, setDraftTitle] = useState<string>("");
```

Click on a row's title → `setEditingId(item.id); setDraftTitle(item.title ?? deriveTitle(item))`. Click outside or Esc → `setEditingId(null)`. Enter or blur with non-empty draft → `renameMutation.mutate({ id, title: draftTitle })`.

### Step 3 — Derive a default title from the events

A small client-side helper that reads the first user-text part from the persisted event log:

```ts
function deriveDefaultTitle(itemId: string, items: SidebarItem[]): string {
  // If the conversation has events, the server has them. But the sidebar
  // doesn't have them in SidebarItem. Two options:
  //   (a) fetch /api/conversations/:id/events on click of the rename UI
  //   (b) extend the sidebar's fetch to include the first 1-2 events
  // For v1, (a): fetch on demand. The latency is hidden by the modal
  // opening + the user typing latency budget.
  return "";  // placeholder — Step 3.1 below
}
```

#### 3.1 — Lazy fetch the events when the rename UI opens

The cleanest way: when the user clicks the title to rename, fetch `GET /api/conversations/:id/events?limit=1` and look at the first `message.received` event's `data.parts[0].text`. Pre-fill the input with that. If the fetch fails or there are no events, pre-fill empty.

This adds a single `useQuery({ enabled: editingId === id })` to drive the prefetch. Cache the result in TanStack Query keyed by `[conversation, id, "rename-prefill"]`.

### Step 4 — Inline edit UI

Replace the title `<span>` with one of:

```tsx
{editingId === item.id ? (
  <Input
    autoFocus
    value={draftTitle}
    onChange={(e) => setDraftTitle(e.target.value)}
    onBlur={() => commitRename(item.id)}
    onKeyDown={(e) => {
      if (e.key === "Enter") commitRename(item.id);
      if (e.key === "Escape") setEditingId(null);
    }}
  />
) : (
  <button onClick={() => startEditing(item)}>
    {item.title ?? item.preview ?? "New chat"}
  </button>
)}
```

`Input` is the shadcn primitive (`components/ui/input.tsx` — already installed). `commitRename` is the wrapper that calls the mutation only if `draftTitle.trim()` is non-empty.

### Step 5 — Visual affordance

A pencil icon (lucide `Pencil`) that appears on hover, positioned to the right of the title. CSS-only via `group-hover:opacity-100 opacity-0` on the icon. Click on the icon = click on the title (same handler).

For the active conversation, the pencil is always visible (not just on hover) so the user can rename the current chat without hovering.

### Step 6 — Handle the "current conversation" case

If the user renames the chat they're currently viewing (`pathname === /?c=<id>`), the chat header in the layout (where the agent name is) is unaffected — the agent name is hardcoded "eve-template" today, not the conversation title. So renaming the conversation doesn't break the page.

If we later surface the conversation title in the chat header, the rename mutation's optimistic update + `setQueryData` will be picked up by the page (which already reads the conversation via the page-level fetch). No additional wiring needed.

## Open questions

These will be resolved during implementation, not at spec time:

- **Auto-derive server-side vs client-side?** — recommendation: client-side (one-time fetch on click of the rename UI, hidden by the modal-open latency).
- **Empty title UX?** — recommendation: revert to "New chat" placeholder when the draft is empty after trim. The PATCH endpoint accepts empty strings and the SidebarItem render falls back.
- **Show a "Saved!" toast?** — recommendation: skip in v1. The optimistic cache update is the visual confirmation.
- **Keyboard shortcut to start editing?** — recommendation: skip in v1. Hover-revealed pencil is enough.

## Risks / edge cases

- **Race between rename and turn-end events**: if the user renames while a turn is in flight, the PATCH and the events POST race. The `updatedAt` in the conversation is bumped twice. No data corruption (separate columns), just a possible re-sort. Acceptable.
- **Optimistic update with rollback on slow network**: if the PATCH takes >2s, the user might click again, queueing two renames. The cache debounces by `queryKey` but the network is unbounded. Mitigation: disable the input while the mutation is in flight (`disabled={renameMutation.isPending}`).
- **Very long titles**: the `title` column is `text` (unbounded in Postgres). The sidebar's `truncate` class already handles display. The rename input should have `maxLength={120}` as a soft limit to prevent abuse.
- **Empty title from server**: if the user submits an empty string, the PATCH accepts it (current code). The sidebar shows "New chat". This is the intended behavior. Documented in AC.
