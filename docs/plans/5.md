## TL;DR

A signed-in user gets a left-rail sidebar in the chat view listing all their non-archived conversations (most-recent first, pinned on top), with a `+ New chat` action and per-thread links; each conversation durably persists its `eve_session_id` + continuation token + event log so the user can refresh, navigate away, and resume the same eve session hours later.

---

## Context

The current chat UI (`app/(authenticated)/page.tsx`) renders a single conversation backed by one in-memory `useEveAgent()` hook. There is no list of past threads, no way to start a "new chat" without losing the current one, and no way to reload the page and keep working — `eve_session_id` lives only in `useState`. The eve runtime (`agent/`) stores conversation history durably server-side but does not enumerate sessions (verified against `node_modules/eve/dist/src/protocol/routes.d.ts` — no `GET /sessions` route exists), so the index must live in **our** database, scoped by the better-auth `user.id`. This issue tracks the schema, the 8 API routes, the sidebar UI built on the official shadcn `sidebar` primitive, the nested route group that keeps the sidebar off of `settings/`, and the chat wiring (`onFinish`/`onEvent` persistence with batching + idempotency).

This is **independent of #3** (better-auth multi-device session revoke) — both touch the word "session" but live in different layers (`auth.session` cookie row vs `conversation` chat thread).

### Step 3 — Add the chat schema

**File:** `db/schema/chat.ts` (create).

Two tables (matches the issue AC exactly):

- `conversation` — `id`, `userId` (FK → `user.id`, cascade), `eveSessionId`, `eveContinuationToken`, `eveStreamIndex`, `title`, `preview`, `messageCount`, `pinned`, `archivedAt`, `createdAt`, `updatedAt`. Indexes `(userId, updatedAt DESC)` and `(userId, archivedAt)`.
- `conversationEvent` — `id`, `conversationId` (FK → `conversation.id`, cascade), `sequence` (per-turn monotonic), `eventId` (id from eve stream envelope for dedupe), `type` (whitelisted string), `payload` (JSONB), `byteSize` (int), `createdAt`. `UNIQUE (conversationId, eventId)` to absorb duplicates. Index `(conversationId, sequence)`.

Implementation detail (decided in spec, not left open): use `text` ids from `nanoid` (5.x API: `nanoid()`), store `timestamp` with `withTimezone: true, mode: 'date'`, use `boolean` with `notNull()` for `pinned`.

### Step 4 — Shared helpers (`requireUser`, `conversations`)

**Files:** `lib/require-user.ts` (create), `lib/conversations.ts` (create).

`lib/require-user.ts`:
```ts
import { auth } from "@/lib/auth";
import { headers } from "next/headers";
import { NextResponse } from "next/server";

// `auth.api.getSession` returns `{ session, user } | null` (verified against
// better-auth 1.7.0-rc.1 source: node_modules/better-auth/dist/api/routes/session.d.mts
// line 9-47). Both are typed against the user/session model defined in
// `db/schema/auth.ts`, augmented with plugin-specific fields (e.g. the admin
// plugin adds `role`, `banned`, `banReason`, `banExpires` on user and
// `impersonatedBy` on session).
export async function requireUser() {
  const result = await auth.api.getSession({ headers: await headers() });
  if (!result) {
    return { error: NextResponse.json({ error: "unauthorized" }, { status: 401 }) };
  }
  return { userId: result.user.id };
}
```

`lib/conversations.ts`: thin functions over Drizzle, each accepting `(userId, ...)` and **never trusting caller-supplied userId** (the route always passes the verified one). They mirror the AC list and the routes in Step 5: `listConversations(userId, { limit })`, `getConversation(userId, id)`, `createConversation(userId)`, `updateConversation(userId, id, patch)`, `deleteConversation(userId, id)`, `appendEvents(userId, id, events)` (filters inserts > 256 KB), `updateEveState(userId, id, { sessionId, continuationToken, streamIndex })`, `listEvents(userId, id, { since?, limit })`. Plus a small `eventTypeWhitelist` Set of valid eve event types (read from `eve/dist/src/protocol/message.ts` at build time, hard-coded for simplicity) — events with unknown `type` are dropped and counted in a debug log.

**Why this layer:** every API route becomes 5–15 lines, the type contract lives in one file, and `requireUser` is the only place that touches `auth.api.getSession`, which keeps the cookie touch idempotent and easy to memoize later.

#### 4.1 — Event-type whitelist

The whitelist (declared in `lib/events.ts` and reused by both the route and the client-side batcher for schema validation) is the union of all `type:` strings on the 28 members of `HandleMessageStreamEvent` in `node_modules/eve/dist/src/protocol/message.d.ts` (line 523). Authoritative set (verified 2026-07-17 by reading the source):

```
session.started
session.waiting
session.completed
session.failed
turn.started
turn.completed
turn.failed
turn.cancelled
step.started
step.completed
step.failed
message.received
message.appended
message.completed
reasoning.appended
reasoning.completed
actions.requested
action.result
input.requested
authorization.required
authorization.completed
result.completed
compaction.requested
compaction.completed
subagent.started
subagent.called
subagent.completed
subagent.child-event
```

(Other discriminants may appear — for example `result.completed.data.result` carries a typed payload, but the `type` string itself is fixed in the union.) Anything else is treated as a protocol drift: dropped on the floor, logged once at level `warn` with the offending `type` and the conversation id. The schema in `conversation_event.type` is `text NOT NULL` (not an enum) so future eve versions can add new event types without a migration.

### Step 5 — API routes

Eight route handlers, all built on top of `requireUser()` + `conversations.*`. Each route file starts with the same three lines.

**Files:**
- `app/api/conversations/route.ts` — `GET` (list, 50, archivedAt IS NULL), `POST` (create, returns `id`).
- `app/api/conversations/[id]/route.ts` — `GET` (meta + first 100 events via `?since=0&limit=100`), `PATCH` (rename / pin / archive via whitelist), `DELETE` (cascade).
- `app/api/conversations/[id]/eve-state/route.ts` — `POST`, body `{ sessionId?, continuationToken?, streamIndex }`. Rejects when `eveStreamIndex` regresses (LWW) unless `force: true` (used by reconnect).
- `app/api/conversations/[id]/events/route.ts` — `POST` accepts `{ events: [{ id?, type, sequence, payload }] }`, returns `{ accepted, rejected: [{ index, reason }] }`. `GET ?since=<seq>&limit=<n>` returns `{ events: [...], hasMore }`. Default `limit=100`, cap `500`. Reject (per event) if `byteSize > 256 * 1024` (computed from stringified `payload`).

#### 5.1 — Idempotency

Two layers:
1. `UNIQUE (conversation_id, event_id)` — duplicates from a retry of `POST /events` hit a constraint violation and are absorbed (`ON CONFLICT DO NOTHING`).
2. `eventId` is eve's per-event id (extracted from each `HandleMessageStreamEvent.id`). If absent in the payload, derive from `nanoid()` on the client (rare; eve events always have one).

#### 5.2 — Rate limit / abuse

Documented but **not implemented in v1**. Note in Risks. Smoke tests stay < 1k events; production-side abuse is a follow-up.

#### 5.3 — CSRF mitigation

Origin header check (`request.headers.get("origin")` must match `NEXT_PUBLIC_BETTER_AUTH_URL` or be `null` for same-origin fetch). SameSite=lax cookie is already set by better-auth. We add the origin guard in a per-route wrapper:

```ts
function assertSameOrigin(req: Request) {
  const origin = req.headers.get("origin");
  const expected = new URL(process.env.NEXT_PUBLIC_BETTER_AUTH_URL ?? req.url).origin;
  if (origin && origin !== expected) throw new Response(null, { status: 403 });
}
```

`betterAuthFn` (eve channel) already restricts who can talk to eve; this is a second layer on the Next.js side.

### Step 8 — Page router + `useEveAgent` wiring

**File:** `app/(authenticated)/(with-sidebar)/page.tsx` (create, replacing `app/(authenticated)/page.tsx` which is then deleted).

Server component:
1. `const session = await auth.api.getSession({ headers: await headers() });` (already implicit via `(authenticated)/layout.tsx`).
2. Read `?c=` from the URL.
3. If absent: query `listConversations(userId, { limit: 1 })`. If found → `redirect(/?c=<most-recent-id>)`. If not → `POST /api/conversations` server-side (or render `<NewChatInitiator />` client-side that auto-creates).
4. Call `getConversation(userId, id)` and `listEvents(userId, id, { limit: 100 })`.
5. Hand `conversationId`, `initialSession` (from `eveSessionId` / `eveContinuationToken` / `eveStreamIndex`), and `initialEvents` to `<AgentChat conversationId initialSession initialEvents />`.

**File:** `app/_components/agent-chat.tsx` (edit).

```tsx
"use client";
const agent = useEveAgent({
  key: conversationId,                     // remount when switching threads
  initialSession,
  initialEvents,
  prepareSend: () => ({ message, clientContext: { conversationId } }),
  onEvent: (e) => batcher.push(e),
  // onSessionChange fires in the `finally` of `send()` (verified by reading
  // eve/dist/src/client/eve-agent-store.js) — that is, AFTER each turn
  // completes (success, error, or abort) and in `reset()`. It does NOT fire
  // mid-turn on individual events. This is still the right hook for our PATCH
  // because:
  //   - On the very first user message, the new turn ends → onSessionChange
  //     fires with the freshly-assigned sessionId. We persist it server-side.
  //   - On every subsequent turn, the new continuationToken (from
  //     session.waiting) is reflected in `session.continuationToken` and we
  //     persist it on the same hook fire.
  //   - Initial render of a conversation: `initialSession` is already
  //     populated from the DB by the server component, so the hook does not
  //     need to do anything on mount.
  //   - Skip the fire when the cursor has not actually changed since the last
  //     PATCH (e.g. abort with no events received).
  onSessionChange: (session) => {
    if (!session.sessionId) return;        // guard: skip until the server assigns one
    if (session.sessionId === lastPersistedSessionId &&
        session.continuationToken === lastPersistedContinuationToken &&
        session.streamIndex === lastPersistedStreamIndex) {
      return;                              // no-op: cursor unchanged since last PATCH
    }
    fetch(`/api/conversations/${conversationId}/eve-state`, {
      method: "POST",
      body: JSON.stringify({
        sessionId: session.sessionId,
        continuationToken: session.continuationToken,
        streamIndex: session.streamIndex,
      }),
    });
    lastPersistedSessionId = session.sessionId;
    lastPersistedContinuationToken = session.continuationToken;
    lastPersistedStreamIndex = session.streamIndex;
  },
  onFinish: (snapshot) => {
    // onFinish is reserved for client-only side-effects (analytics, toast). Not
    // used for eve-state here — see onSessionChange above.
  },
});
```

#### 8.1 — Client-side batcher

**File:** `app/_components/agent-chat-batcher.ts` (create).

```ts
// Flushes every 500ms OR when 50 events accumulate.
// On flush, POST { events: [...] } to /api/conversations/:id/events.
// On 409 (replayed), drop those events.
// On 4xx other than 401/403/409, keep buffer and retry next tick (capped 5 attempts).
class EventBatcher {
  buffer: EveEvent[] = [];
  timer?: number;
  enqueue(e: EveEvent) { ...; this.scheduleFlush(); }
  flush() { ... }
}
```

Why this lives client-side: the wire shape stays small (one POST per ~500ms instead of one per event). Server-side multi-row insert (`db.insert(...).values([...])`) for the cost of one round-trip.

## Open questions

These will be resolved during implementation:

- **Title-source race:** when does `title` get set on first `message.completed`? Easiest: client-side `onEvent` flush that one specific event triggers a separate `PATCH /api/conversations/:id` with the new title. Decision: client does the patch lazily (after the batch flush, before `onFinish` of the turn that produced the first user text). No separate cron.
- **`preview` update cadence:** same as `title` — only on `message.completed`, rolled over (preview = last assistant text). Updates the `conversation_event` table for `byteSize` accounting via the same path.

## Risks / edge cases

- **CI drift on pglite version mismatch** — `pglite` versions can ship new Postgres features not understood by the migration. Mitigation: pin `@electric-sql/pglite` exact, and run `drizzle-kit check` to catch unsupported types.
- **Event growth** — a long session can produce thousands of events. The smoke test stays small; the integration test inserts 1k events and asserts `listEvents` cursor pagination. Spec does **not** include a retention policy yet — flagged for follow-up.
- **Multi-tab race on the same `conversationId`** — two `useEveAgent` instances can race on `appendEvents`. The `UNIQUE (conversation_id, event_id)` constraint absorbs duplicates; an out-of-order `eve-stream-index` write loses to the higher value (LWW). Documented in `lib/conversations.ts` header. Surfaced as a user-visible risk in the issue Risks section.
- **Origin check** — `assertSameOrigin` blocks browser cross-origin POSTs. CLI/curl traffic unaffected. Test: smoke test of the wrapper verifies behavior with a fake origin header.
- **Drizzle barrel index** — switching `drizzle.config.ts` from a single-file pointer to `index.ts` requires re-running `drizzle-kit generate` for the new schema. If missed, migrations don't see the new tables. CI test catches it (the integration test runs migrations).
- **`shadcn` CLI version drift** — covered by the issue Risk; check `components.json` before/after. The CLI is always `latest`, but we manually pin by copying the file in and reverting unintentional diffs.
- **Better-auth `user.id` foreign key** — the existing `db/schema/auth.ts#user.id` is `text`. Our `conversation.userId` is also `text`. Drizzle generates the FK correctly. Verify in the generated migration.
