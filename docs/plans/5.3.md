## TL;DR

Add a delete action to the sidebar conversation list — hover-revealed trash icon on each row → click → shadcn `AlertDialog` confirmation → DELETE the conversation → remove from the local cache. The DELETE endpoint already exists (`DELETE /api/conversations/:id` cascades events), so the work is a small client-side mutation + dialog + a redirect path when the user deletes the chat they're currently viewing. The `archivedAt` column already supports soft delete as a follow-up; v1 ships hard delete only.

---

## Context

The data layer for deletion is in place: `app/api/conversations/[id]/route.ts#DELETE` calls `deleteConversation(userId, id)` in `lib/conversations.ts`, which does a hard delete with `ON DELETE CASCADE` on `conversation_event`. The sidebar's `SidebarItem` already supports removal from the cache via TanStack Query's `setQueryData`. What's missing is the UI: there's no way for the user to clean up old chats from the sidebar.

The schema also supports soft delete via the `archivedAt` column (set via `updateConversation({ archivedAt: now() })`), but the existing list query filters on `archivedAt IS NULL`. v1 uses hard delete; soft delete / archive is a follow-up.

---

## Scope

**IN:**
- Trash icon in the sidebar item, hover-revealed (always visible for the active item, like the rename pencil)
- Click → shadcn `AlertDialog` with a typed-confirmation requirement ("Delete" must be typed) to prevent misclicks
- DELETE call to `/api/conversations/:id` on confirm
- Optimistic removal from the TanStack Query cache (the row disappears from the sidebar immediately)
- Error path: re-add the row to the cache if the DELETE fails, show a toast/inline error
- Redirect path: if the user deletes the chat they're currently viewing (`pathname === /?c=<id>`), navigate to `/` (which plan 5.1's Step 3 redirect to most-recent kicks in)

**OUT (deferred to follow-ups):**
- Soft delete / archive (the `archivedAt` column is there, the wire is PATCH; a separate "Archive" button lands in 5.4 or later)
- Bulk delete
- Restore deleted (would require a trash bin or undelete endpoint)
- Delete from the chat page itself (delete via a menu in the chat header) — sidebar only in v1

## Step-by-step implementation

### Step 1 — Extract the confirmation dialog

A new client component `app/_components/delete-conversation-dialog.tsx`:

```tsx
"use client";

import { useState } from "react";
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
  AlertDialogTrigger,
} from "@/components/ui/alert-dialog";
import { Input } from "@/components/ui/input";
import { Label } from "@/components/ui/label";
import { Button } from "@/components/ui/button";

export function DeleteConversationDialog({
  conversationTitle,
  onConfirm,
  trigger,
}: {
  conversationTitle: string;
  onConfirm: () => void | Promise<void>;
  trigger: React.ReactNode;
}) {
  const [typed, setTyped] = useState("");
  const matches = typed === "Delete";
  return (
    <AlertDialog>
      <AlertDialogTrigger asChild>{trigger}</AlertDialogTrigger>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>Delete conversation?</AlertDialogTitle>
          <AlertDialogDescription>
            This permanently deletes <strong>"{conversationTitle}"</strong> and
            all of its messages. The eve-side session is left to expire on
            its own. This cannot be undone.
          </AlertDialogDescription>
        </AlertDialogHeader>
        <div className="space-y-2">
          <Label htmlFor="confirm-delete">
            Type <span className="font-mono">Delete</span> to confirm
          </Label>
          <Input
            id="confirm-delete"
            autoFocus
            value={typed}
            onChange={(e) => setTyped(e.target.value)}
          />
        </div>
        <AlertDialogFooter>
          <AlertDialogCancel>Cancel</AlertDialogCancel>
          <AlertDialogAction asChild>
            <Button
              variant="destructive"
              disabled={!matches}
              onClick={onConfirm}
            >
              Delete
            </Button>
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );
}
```

The `typed === "Delete"` check is the friction guard. Users have to type the word to enable the destructive button. A simple "Are you sure?" dialog is too easy to misclick; typed confirmation is the industry standard for destructive actions in chat apps (Notion, Linear, etc.).

### Step 2 — Add the trash icon to the sidebar

In `app/(authenticated)/_components/sidebar.tsx`, alongside the existing pin emoji:

```tsx
<SidebarMenuAction disabled>📌</SidebarMenuAction>  // existing pin
<DeleteConversationDialog
  conversationTitle={label}
  onConfirm={() => deleteMutation.mutate(item.id)}
  trigger={
    <SidebarMenuAction className="opacity-0 group-hover/menu-item:opacity-100">
      🗑
    </SidebarMenuAction>
  }
/>
```

The `group-hover/menu-item:opacity-100` pattern requires the `SidebarMenuItem` to have `group` set (the shadcn primitive already does this). For the active item, the trash icon should be always visible — handled by overriding the opacity in the active variant.

### Step 3 — Add the `deleteMutation`

Same optimistic pattern as `createMutation` and `renameMutation`:

```ts
const deleteMutation = useMutation({
  mutationFn: async (id: string) => {
    const res = await fetch(`/api/conversations/${id}`, { method: "DELETE" });
    if (!res.ok) throw new Error("Failed to delete");
    return id;
  },
  onMutate: async (id) => {
    await qc.cancelQueries({ queryKey: CONVERSATIONS_QUERY_KEY });
    const previous = qc.getQueryData<SidebarItem[]>(CONVERSATIONS_QUERY_KEY);
    const removed = previous?.find((i) => i.id === id) ?? null;
    qc.setQueryData<SidebarItem[]>(CONVERSATIONS_QUERY_KEY, (old) =>
      (old ?? []).filter((i) => i.id !== id),
    );
    return { previous, removed };
  },
  onError: (_err, _vars, ctx) => {
    if (ctx?.previous) qc.setQueryData(CONVERSATIONS_QUERY_KEY, ctx.previous);
    if (ctx?.removed) {
      // surface a toast or inline error
    }
  },
  onSuccess: (id) => {
    // If the deleted chat is the one currently being viewed, navigate away.
    if (pathname === `/?c=${id}`) {
      router.push("/");
    }
  },
});
```

The `onSuccess` redirect is critical: if the user deletes the current chat, they land on `/`, which plan 5.1's Step 3 redirects to the most-recent remaining conversation (or the empty state).

### Step 4 — Toast for errors (optional, defer to v1.1)

For v1, a `console.error` on failure is sufficient. The cache rollback restores the row visually. The user will see the row re-appear if the DELETE failed.

A proper toast (shadcn `sonner` or `useToast`) is a follow-up. Adding it now expands the scope.

## Open questions

These will be resolved during implementation, not at spec time:

- **Typed confirmation vs. "Are you sure?" button?** — recommendation: typed confirmation. Industry standard for destructive actions; trivial to implement.
- **Where to trigger the dialog?** — recommendation: a hover-revealed trash icon on the right edge of each row, mirroring the pin action.
- **Toast on error?** — recommendation: skip in v1. The cache rollback is enough visual feedback.
- **Bulk delete?** — recommendation: out of scope.

## Risks / edge cases

- **Deleting the current conversation while a turn is in flight**: the eve server continues the turn (the server has its own session), but the UI is gone. When the user navigates back, the conversation won't exist. Mitigation: the eve server will GC the session after the turn completes (eve's normal flow). No client-side issue.
- **Optimistic removal with slow network**: if the DELETE takes >2s, the user might click delete on a second chat. The cache has already removed the first; the second mutation runs in parallel. The first might fail and roll back; the row reappears. Acceptable.
- **Race between delete and last-event POST**: if the user deletes a chat while the batcher is flushing, the DELETE removes the conversation row, but the events POST arrives after. The events POST then either:
  - Fails the `getConversation` ownership check (returns 404 from the route) — clean
  - Tries to insert into a deleted conversation — FK violation, route returns 500
  This is acceptable: the user wants the chat gone, and any subsequent server-side activity on it is wasted work but harmless.
- **Sidebar orphan icon**: the existing pin emoji is rendered unconditionally for pinned items. If the delete icon is also rendered unconditionally, the row gets crowded. Recommendation: the delete icon replaces the pin (or sits next to it) and is hidden on small screens.
