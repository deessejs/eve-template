## TL;DR

When the user sends their first message in a chat and the agent finishes the first turn, fire a server-side `generateText` call against the same `minimax` model the agent uses, and PATCH the conversation with the result. The sidebar updates with the new title via the existing TanStack Query cache. The user can still override manually via the existing rename dialog (the auto-title route is a no-op if `title != null`, so manual renames win permanently).

---

## Context

Chats are currently created with `title: null` and the sidebar shows the placeholder "New chat" forever. The user can manually rename via the kebab → Rename dialog (plan 5.2 / commit `ad82e4a`). Real apps (ChatGPT, Claude.ai, Notion AI) auto-generate a title on the first turn so the user has something to recognize in the sidebar immediately. We have everything we need:

- The `ai` SDK (v7.0.29) is installed and exposes `generateText`
- The `vercel-minimax-ai-provider` (v0.0.2) is installed and exported as `minimax` (Anthropic-compatible) — the same model the agent itself uses
- The `agent-chat.tsx` already subscribes to the eve stream via `useEveAgent`'s `onEvent` / `onFinish` callbacks
- The PATCH endpoint already exists (`PATCH /api/conversations/:id` with `{ title }`) and `lib/conversations.ts#updateConversation` is battle-tested

The missing piece is just: a watcher that detects the first `turn.completed` event, extracts the first user-text, and a server route that calls `generateText` and PATCHes the title.

---

## Scope

**IN:**
- New server route `POST /api/conversations/:id/generate-title` (≈60 lines)
- `generateText` call with the `minimax` model and a title-derivation prompt
- Truncation: user message → 500 chars; final title → 80 chars
- Skip if `conversation.title` is already set (manual rename wins)
- Skip if the user message is < 4 chars (avoid "ok", "hi", "1")
- Fallback to a manual truncation of the user message if the model returns empty/whitespace
- Client-side watcher in `agent-chat.tsx` (one-shot, ref-guarded) that POSTs to the route after the first `turn.completed`
- TanStack Query cache update on success so the sidebar reflects the new title
- No-op (silent log) on errors — the title is non-critical, never block the chat
- Smoke test that the new route file exists + integration test that the watcher calls the route

**OUT (deferred to follow-ups):**
- AI-suggested titles on demand (a "Suggest a title" button)
- Re-titling when the conversation diverges from the original topic (some apps re-title after N turns)
- Multi-language detection (we use the same language as the user message; the model handles this naturally)
- Custom title templates per user / per workspace

## Step-by-step implementation

### Step 1 — The server route

`app/api/conversations/[id]/generate-title/route.ts`:

```ts
import { generateText } from "ai";
import { minimax } from "vercel-minimax-ai-provider";
import { NextResponse, type NextRequest } from "next/server";

import { assertSameOrigin, requireUser } from "@/lib/require-user";
import { getConversation, updateConversation } from "@/lib/conversations";

type RouteContext = { params: Promise<{ id: string }> };

const MIN_USER_MESSAGE_LENGTH = 4;
const MAX_USER_MESSAGE_CHARS = 500;
const MAX_TITLE_CHARS = 80;
const FALLBACK_TITLE_CHARS = 60;

const SYSTEM_PROMPT = [
  "Generate a concise title (3-6 words) for the following user message.",
  "The title should be in the same language as the message.",
  "Use sentence case (capitalize only the first word and proper nouns).",
  "Do NOT include punctuation, quotes, or trailing periods.",
  "Output ONLY the title, nothing else.",
].join("\n");

function cleanTitle(raw: string, fallback: string): string {
  let t = raw.trim();
  t = t.replace(/^["'""''`]+|["'""''`]+$/g, ""); // strip quotes
  t = t.replace(/[.!?…]+$/, "");                  // strip trailing punct
  t = t.split(/\r?\n/)[0] ?? t;                   // first line only
  t = t.slice(0, MAX_TITLE_CHARS).trim();
  return t.length >= 1 ? t : fallback;
}

export async function POST(request: NextRequest, ctx: RouteContext) {
  const sameOrigin = assertSameOrigin(request);
  if (sameOrigin) return sameOrigin;
  const auth = await requireUser();
  if ("error" in auth) return auth.error;
  const { id } = await ctx.params;

  const body = (await request.json().catch(() => null)) as
    | { userMessage?: string }
    | null;
  const userMessage = body?.userMessage?.trim() ?? "";
  if (userMessage.length < MIN_USER_MESSAGE_LENGTH) {
    return NextResponse.json(
      { error: "user_message_too_short" },
      { status: 400 },
    );
  }

  // Skip if the user has already set a title manually.
  const existing = await getConversation(auth.userId, id);
  if (!existing) {
    return NextResponse.json({ error: "not_found" }, { status: 404 });
  }
  if (existing.title) {
    return NextResponse.json({ item: existing }); // no-op success
  }

  const truncated = userMessage.slice(0, MAX_USER_MESSAGE_CHARS);
  const fallback = truncated.slice(0, FALLBACK_TITLE_CHARS);

  let generated: string;
  try {
    const { text } = await generateText({
      model: minimax("MiniMax-M3"),
      system: SYSTEM_PROMPT,
      prompt: truncated,
      maxOutputTokens: 50,
      temperature: 0.3,
    });
    generated = cleanTitle(text, fallback);
  } catch (err) {
    // Non-critical: log and return fallback so the user still gets
    // *something* to recognize in the sidebar.
    console.warn("[generate-title] model call failed, using fallback", err);
    generated = fallback;
  }

  const item = await updateConversation(auth.userId, id, { title: generated });
  return NextResponse.json({ item });
}
```

### Step 2 — The client watcher

In `app/_components/agent-chat.tsx`, add a one-shot ref + effect:

```ts
const firstTurnCompletedRef = useRef(false);

useEffect(() => {
  if (firstTurnCompletedRef.current) return;
  if (agent.status !== "ready") return;          // wait for first turn to settle
  if (agent.events.length === 0) return;        // no events = no first turn

  // Find the first user-text part from the message.received events.
  const firstUserText = agent.events.find(
    (e) => e.type === "message.received",
  )?.data?.parts?.find((p) => p.type === "text")?.text;

  if (typeof firstUserText !== "string") return;
  if (firstUserText.trim().length < 4) return;

  firstTurnCompletedRef.current = true;          // one-shot

  if (!conversationId) return;
  void fetch(`/api/conversations/${conversationId}/generate-title`, {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify({ userMessage: firstUserText.slice(0, 500) }),
  })
    .then((res) => (res.ok ? res.json() : null))
    .then((data) => {
      if (data?.item) {
        qc.setQueryData<Array<{ id: string; title: string | null }>>(
          CONVERSATIONS_QUERY_KEY,
          (old) =>
            (old ?? []).map((i) =>
              i.id === conversationId
                ? { ...i, title: data.item.title }
                : i,
            ),
        );
      }
    })
    .catch(() => {
      // non-critical; the title is best-effort
    });
}, [agent.status, agent.events, conversationId, qc]);
```

## Acceptance criteria

- [ ] After the user sends their first message AND the agent completes the first turn, the conversation's `title` is auto-populated by `generateText`
- [ ] The title is 3-6 words, sentence case, no trailing punctuation
- [ ] If the user has manually set a title, the auto-title is a no-op (manual wins)
- [ ] If the user message is < 4 chars, no title is generated
- [ ] If `generateText` fails or returns empty, the route falls back to a 60-char truncation of the user message
- [ ] The sidebar updates with the new title within 500ms of the first turn completing (via TanStack Query optimistic update)
- [ ] Manual rename via the kebab → Rename dialog still works and persists (no overwriting by subsequent auto-titles because the route no-ops when `title` is set)
- [ ] The watcher is a one-shot per conversation (no duplicate POSTs on subsequent turns)
- [ ] Two tabs in the same conversation don't both fire (the `firstTurnCompletedRef` is per-tab, but the server no-ops the second one because `title` is set)
- [ ] `npm test` passes (27+ tests including 1 new smoke check + 1 new integration test that's skipped without `MINIMAX_API_KEY`)
- [ ] `npm run typecheck` and `npm run lint` clean

## Open questions

These will be resolved during implementation, not at spec time:

- **What model to use for title generation?** — recommendation: the same `minimax("MiniMax-M3")` the agent uses. Consistent, no extra config. A small dedicated model would be faster but adds a dep.
- **Re-title on long conversations?** — recommendation: skip in v1. The manual rename is the override. If the topic shifts, the user can rename.
- **Show a "Generating title..." indicator?** — recommendation: skip. The title appears ~500ms after the response; showing a loading state for half a second is more noise than signal.
- **Cache the generated title?** — recommendation: skip. The cost is ~50 tokens per chat; negligible.
- **What if MINIMAX_API_KEY is missing?** — recommendation: the route falls back to the user-message truncation. The chat itself also fails without the key, so this is consistent with the rest of the app's behavior.

---

## Risks / edge cases

- **Latency on the first turn**: `generateText` adds 200-500ms after the response. The user already sees the assistant's reply, so the title arriving slightly later is fine.
- **Cost**: ~50 input tokens + ~10 output tokens per chat. At 1k new chats/day, this is ~$0.01/day. Negligible.
- **Model returns weird output**: the `cleanTitle` function handles the common cases (quotes, trailing punct, newlines, whitespace). Edge cases like unicode emoji in the title are passed through (which is fine — titles can include emoji).
- **Privacy**: the route sees the user's first message. Same auth as the chat itself, so the threat model is identical.
- **Concurrency**: two tabs in the same conversation both fire the watcher; the server's `if (existing.title)` guard makes the second one a no-op. Idempotent.
