# Contributing to `eve-template`

Thanks for your interest in improving this scaffold. This document covers how
to set up a dev environment, the lifecycle a change goes through, and the
conventions to follow so your PR lands cleanly.

---

## What this project is (and isn't)

`eve-template` is a **starter scaffold** for shipping a production-ready
[Vercel `eve`](https://eve.dev) agent with a web chat UI. It bundles the
canonical `agent/` directory, a Next.js 16 web channel, better-auth (login-only),
Drizzle/Postgres durability, and the wiring to deploy both services on Vercel.

What it isn't:

- **Not** a generic Next.js starter. If you don't need the agent runtime,
  you don't need this template.
- **Not** a hosted product. Users clone, install, and run.
- **Not** a published npm package. Releases are git tags + GitHub Releases
  on this repo, not registry publishes.

If your change targets only the user-facing chat UI without touching the
agent, please open the PR against `app/`, `components/`, or `lib/` and tag
`area:web`. If it touches the agent's behavior, prefer `agent/` or
`db/` and tag `area:auth` / `area:database` / `area:app` accordingly.

## Development setup

### Prerequisites

- **Node.js 24.x** — `fnm use 24` (or `nvm use 24`). `package.json#engines`
  pins 24.x.
- **npm 10+** (ships with Node 24).
- **PostgreSQL** — local Docker, Neon, or Vercel Postgres. You'll need the
  connection string before any auth/agent work makes sense.

### Install

```bash
git clone https://github.com/deessejs/eve-template.git
cd eve-template
npm install --legacy-peer-deps   # --legacy-peer-deps is required (see notes below)
cp .env.example .env             # fill in DATABASE_URL, BETTER_AUTH_SECRET, MINIMAX_API_KEY
```

The `--legacy-peer-deps` flag is non-optional: better-auth `1.7.0-rc.1`
pins `jose ^6.1.0` and `kysely ^0.28.17` via `package.json#overrides`, and
npm's strict peer resolver rejects the RC's transitive tree without it.
CI does the same (see `.github/workflows/ci.yml`).

### Run

```bash
npm run dev          # web chat UI at http://localhost:3000
                     # eve runtime at /_eve_internal/eve/*
npm run dev:eve      # the eve TUI/runner alone (no Next.js)
npm run typecheck    # tsc --noEmit -p tsconfig.json
```

> The dev server is yours to start. The `eve-template` project does not
> spawn HTTP servers from coding agents.

### Bootstrap the first admin

```bash
bash scripts/create-admin.sh \
  --email admin@your-domain.com \
  --password 'something-strong-here' \
  --name "Admin"
```

Subsequent users (admin or not):

```bash
node --env-file=.env scripts/create-user.mjs \
  --email alice@your-domain.com \
  --name "Alice" \
  --password 'something-strong-here' \
  --role user
```

See [`README.md`](./README.md) § Authentication for the full sign-in flow.

## Branching model

```
feature/<name> ──PR──> staging ──ship──> release/v<X.Y.Z> ──PR──> main
                                                                   │
                                                            gh release create
                                                            (agent-driven)
```

- **`feature/<name>`** — your branch. Open against `staging`, not `main`.
- **`staging`** — integration branch. Multiple features land here first; the
  web chat and agent are exercised against it before a release.
- **`release/v<X.Y.Z>`** — short-lived branch cherry-picked from selected
  PRs on `staging`, opened against `main` by the `ship` skill.
- **`main`** — production-ready. Each push triggers a release tag and
  GitHub Release, generated by the `release` skill.

> **Never push to `main` directly.** Always go through a `release/v*`
> branch or the `ship` skill.

## Workflow: spec → implement → PR → release

This repo ships **7 Claude agent skills** that orchestrate the full change
lifecycle. If you use Claude Code, prefer the skills — they enforce the
Definition of Done and consistency rules for you. Manual equivalents are
documented below.

| Stage | Skill | Manual equivalent |
|---|---|---|
| Discovery | — | Read `node_modules/eve/docs/` and the relevant `docs/learnings/` page |
| Specification | [`spec`](./.claude/skills/spec/SKILL.md) | Write a spec doc in `docs/plans/` |
| Triage | [`triage`](./.claude/skills/triage/SKILL.md) | Verify, deduplicate, label |
| Implementation | [`implement`](./.claude/skills/implement/SKILL.md) | Branch off `staging`, commit, push |
| Review | [`review-pr`](./.claude/skills/review-pr/SKILL.md) | Open a draft PR, ask for review |
| Open PR | [`create-pr`](./.claude/skills/create-pr/SKILL.md) | Open via GitHub UI using `PULL_REQUEST_TEMPLATE.md` |
| Issue creation | [`create-issue`](./.claude/skills/create-issue/SKILL.md) | Use `.github/ISSUE_TEMPLATE/*.yml` |
| Release | [`ship`](./.claude/skills/ship/SKILL.md) + [`release`](./.claude/skills/release/SKILL.md) | Cherry-pick, merge `release/v*` to `main`, `gh release create` |

### Definition of Done

A PR is "done" when **all** of these hold:

- [ ] All acceptance criteria from the spec are met
- [ ] `npm run typecheck` passes
- [ ] `npm run build` passes
- [ ] Tests pass (if any exist; see "Tests" below)
- [ ] Lint passes (if configured)
- [ ] Relevant docs are updated (`README.md`, `docs/`, inline comments)
- [ ] No regressions introduced
- [ ] The implementation stays within the spec's `Scope IN` / `Scope OUT`
- [ ] A `.changeset/<name>.md` file is included in the PR with the correct
      version type (`patch` | `minor` | `major`)

## Coding conventions

### TypeScript

- `tsconfig.json` runs in `strict` mode. Don't loosen it for a single file —
  fix the types.
- Prefer `import type` for type-only imports.
- Public function signatures get explicit return types on the boundary;
  internal helpers can rely on inference.
- Use `zod` (4.x) for runtime validation of tool inputs and external data.

### Directory layout

- `agent/` — eve-managed files (instructions, tools, skills, channels, subagents).
  Adding a new file to the right subdir is registration; nothing else.
  - `agent/tools/*.ts` → tool name = filename
  - `agent/skills/*.md` → skill name = filename
- `app/` — Next.js App Router pages. Route groups in `(parens)` are layout-only.
- `components/` — shared UI primitives (shadcn/ui) and AI Elements.
- `lib/` — framework-agnostic helpers (`auth`, `auth-client`, `eve-auth`, `utils`).
- `db/` — Drizzle schema, migrations, and the connection pool.
- `docs/` — internal learnings and plans. Not user-facing unless explicitly linked.

### Auth

This template is **login-only**. The sign-up endpoint is closed in
`lib/auth.ts` (`disableSignUp: true` + before-hook). Don't re-open it from a
plugin without explicit discussion on the issue first.

better-auth is the default auth stack; Clerk / Auth.js / Auth0 are alternatives
only if an issue asks for them.

### Filename style

- React components: `kebab-case.tsx`
- Hooks / utilities: `kebab-case.ts`
- Drizzle schema tables: singular nouns (`user`, `session`, `account`)
- SQL migrations: `NNNN_<snake_case>.sql`

## Tests

There is no test suite yet. Adding one is tracked but not blocking. If you
add tests:

- Hit a real Postgres (local Docker or a throwaway Neon branch). Don't mock
  the database in integration tests — earlier this masked a broken migration
  in a sister project and the rot only surfaced in prod.
- Smoke tests on `lib/auth.ts` and `agent/agent.ts` exports are enough to
  start; we don't need a full coverage target on day one.
- When you add a test runner (Vitest is the lightest), wire it into
  `.github/workflows/ci.yml` as a fourth job.

## Commit messages

Conventional commits, in the imperative mood:

```
<type>(<scope>): <subject>

<body>

<footer>
```

Common types in this repo: `feat`, `fix`, `chore`, `docs`, `refactor`,
`test`, `ci`, `build`. Scope is one of: `auth`, `db`, `agent`, `web`,
`ui`, `docs`, `release`.

The `ship` skill parses commits to generate the changelog, so keep
subjects specific and present tense ("add", not "added").

## Pull requests

- **Always** target `staging`, never `main`.
- **Always** include a `.changeset/<name>.md` with the correct bump type.
- Fill in the [`PULL_REQUEST_TEMPLATE.md`](./.github/PULL_REQUEST_TEMPLATE.md).
  The agent-generated template is richer — if you used the `create-pr` skill,
  paste its body verbatim.
- Reference the issue: `Part of #N` or `Closes #N`.
- One logical change per PR. If your diff is >400 lines, consider splitting
  (the `create-pr` skill will warn you above that threshold).
- A PR must be reviewed and CI green before `ship` can pick it.

## Reporting bugs

Use the [bug report template](./.github/ISSUE_TEMPLATE/bug_report.yml). Include:

- Steps to reproduce
- Expected vs actual behavior
- Logs / error output
- The `🎯 Area` label that matches the affected code path

For feature requests, use the [feature request template](./.github/ISSUE_TEMPLATE/feature_request.yml).
For security issues, **do not** file a public issue — see [`SECURITY.md`](./SECURITY.md).
